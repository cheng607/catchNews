## 1. 快速上手

### 1.1 环境要求

| 工具 | 版本 | 用途 |
|------|------|------|
| Python | 3.11+ | 后端运行时 |
| Node.js | 18+ | 前端构建 |
| pnpm / npm | 最新 | 前端包管理 |
| Docker + Docker Compose | 最新 | 一键部署 |
| Git | 2.30+ | 版本控制 |

### 1.2 本地启动（开发模式）

```bash
# 1. 克隆仓库
git clone <repo-url> && cd catchNews

# 2. 后端
cd backend/personal
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env               # 按需修改
uvicorn app.main:app --reload --port 8000

# 3. 前端（另一个终端）
cd frontend/personal
pnpm install                       # 或 npm install
cp .env.example .env.local
pnpm dev                           # http://localhost:3000

# 4. 验证
curl http://localhost:8000/api/v1/health
```

> **注意**：后端运行需设置 `PYTHONPATH`，确保 `core` 包可被导入：
> ```bash
> # Linux/macOS
> export PYTHONPATH=$PWD/../..:$PWD
> # Windows PowerShell
> $env:PYTHONPATH="$PWD\..\..\;$PWD"
> ```

### 1.3 Docker 启动

```bash
cd docker/personal
docker compose up --build
# 后端: http://localhost:8000/docs
# 前端: http://localhost:3000
```

### 2.1 代码层级与职责

| 层级 | 路径 | 职责 | 依赖方向 |
|------|------|------|----------|
| **共享核心** | `backend/core/` | 采集器、数据模型、schema、URL 校验、配置基类 | 无外部依赖 |
| **个人版应用** | `backend/personal/app/` | FastAPI 入口、API 路由、业务逻辑、调度、数据库 | 依赖 core |
| **前端** | `frontend/personal/` | Next.js 页面、UI 组件、API 调用 | 依赖后端 API |

**核心原则**：`core` 不依赖 `personal`，`personal` 单向导入 `core`。

## 3. Git 工作流

### 3.1 分支策略

```
main ────────●────────────●────────────● （稳定版本，里程碑合入）
              \          /
dev ───●───●───●───●───●───●───●──── （日常开发主线）
        \     / \     /
feature/ ●──● fix/ ●──●           （功能/修复分支）
```

| 分支 | 命名 | 来源 | 合入目标 | 说明 |
|------|------|------|----------|------|
| `main` | — | — | — | 稳定可发布版本，仅通过 `dev` 合入 |
| `dev` | — | `main` | `main` | 日常开发集成，所有功能先合到这里 |
| `feature/xxx` | `feature/M0-github-collector` | `dev` | `dev` | 功能开发，用里程碑编号前缀 |
| `fix/xxx` | `fix/weibo-parse-error` | `dev` | `dev` | Bug 修复 |
| `hotfix/xxx` | `hotfix/link-validator-crash` | `main` | `main` + `dev` | 线上紧急修复 |

### 3.2 日常开发流程

```bash
# 1. 从 dev 拉功能分支
git checkout dev
git pull origin dev
git checkout -b feature/M0-github-trending

# 2. 开发 & 提交（小步提交，中文描述）
git add .
git commit -m "feat: 实现 GitHub Trending 页面解析"
git commit -m "feat: 补全 Star 指标 API 调用"
git commit -m "test: 添加 GitHub Trending mock 测试"

# 3. 推送 & 合入
git push origin feature/M0-github-trending
# 创建 PR → dev，自测通过后合入

# 4. 里程碑完成后 dev → main
git checkout main
git merge dev --no-ff -m "chore: 合入 M0 里程碑"
git tag v0.1.0
```

### 3.3 Commit 消息规范（中文）

**格式**：

```
<类型>: <中文简要描述>

[可选正文：补充说明 why / 技术决策]

[可选关联：关联 #issue 或里程碑]
```

- **类型**：英文
- **中文简要描述**：必填，用中文概括本次变更

**类型**（type）：

| 类型 | 含义 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 实现百度热搜采集` |
| `fix` | 修复缺陷 | `fix: 修复 hot-items 空数组返回 500` |
| `refactor` | 重构（不改变外部行为） | `refactor: 抽取 BaseCollector 公共方法` |
| `docs` | 文档变更 | `docs: 补充本地启动说明` |
| `test` | 测试相关 | `test: 添加微博采集器 mock 测试` |
| `chore` | 构建/工具/依赖/CI | `chore: 升级 httpx 到 0.28.0` |
| `style` | 代码格式（不影响逻辑） | `style: ruff format 格式化` |
| `perf` | 性能优化 | `perf: 添加 hot_items 查询索引` |

**完整示例**：

```
feat: 实现 GitHub Trending 页面解析

- 支持 daily / weekly 两个维度
- 使用 selectolax 解析 HTML，提取 repo 名、语言、Star 数
- 异常时返回空列表，不阻塞其他采集器

关联: M0 里程碑
```

```
fix: 修复 track=all 时返回空列表

原因：查询条件错误地将 "all" 作为 track 值过滤
修复：当 track=all 时跳过 track 过滤条件
```

## 4. 开发里程碑详解

### 4.1 M0 — GitHub 闭环（第 1 周）

**目标**：技术向最小可用产品，从采集到展示完整跑通。

```
开发顺序（建议串行推进）：

  ① 后端骨架     →  ② GitHub 采集器  →  ③ 数据入库
       ↓                                    ↓
  ④ API 接口     →  ⑤ 前端 GitHub 卡片  →  ⑥ Docker 打通
```

| # | 任务 | 涉及文件 | 产出 | 验收 |
|---|------|----------|------|------|
| ① | 补全后端骨架：路由注册、DB 初始化、调度启动 | `personal/app/main.py` | `docker compose up` 后端启动无报错 | health 接口返回 ok |
| ② | 实现 GitHub Trending 采集器 | `core/collectors/github_trending.py` | 返回 ≥10 条 RawItem | 单元测试通过 |
| ③ | Star 指标补全 + 数据入库 | `core/collectors/github_stars.py`, `personal/app/services/` | hot_items 表有数据 | DB 查询验证 |
| ④ | GET /hot-items?track=tech API | `personal/app/api/hot_items.py` | JSON 响应 | OpenAPI 文档 + curl 测试 |
| ⑤ | 前端 GitHub 卡片 | `frontend/personal/` | 列表渲染 + 跳转 | 点击跳转 github.com |
| ⑥ | Docker Compose 联调 | `docker/personal/docker-compose.yml` | 一键启动 | `docker compose up` 完整可用 |

**M0 验收清单**：

- [ ] GitHub Trending daily + weekly 至少两维度
- [ ] 每条展示 Star 总数 + 增量
- [ ] 点击跳转 github.com 官方仓库页
- [ ] 数据持久化，重启后不丢失
- [ ] `docker compose up` 一键启动

### 4.2 M1 — 娱乐榜 + 导航（第 2 周）

## 5. 后端开发指南

### 5.1 新增采集器的标准步骤

```
1. 在 core/collectors/ 新建 <platform>.py
2. 继承 BaseCollector，实现 fetch() -> list[RawItem]
3. 在 personal/app/scheduler.py 注册定时任务
4. 在 backend/tests/core/ 新增 test_<platform>.py（mock HTTP）
5. 更新 core/link_validator.py 白名单（如有新域名）
```

示例骨架：

```python
# core/collectors/hackernews.py
from core.collectors.base import BaseCollector, RawItem

class HackerNewsCollector(BaseCollector):
    platform = "hackernews"
    track = "tech"

    async def fetch(self) -> list[RawItem]:
        # 实现采集逻辑
        ...
```

### 5.2 新增 API 接口的标准步骤

```
1. 在 personal/app/api/ 新建或修改路由文件
2. 在 personal/app/main.py 注册 router
3. Service 层逻辑放 personal/app/services/
4. 请求/响应 Schema 放 core/schemas/（如果是通用的）
5. 更新 OpenAPI 文档描述
```

### 5.3 数据库变更

```
1. 修改 core/models/base.py 中的模型定义
2. 当前阶段直接 drop + recreate（开发期数据可丢）
3. 正式上线后引入 Alembic 做迁移
```

### 5.4 提交前检查清单

```bash
# 格式化
ruff format backend/

# Lint 检查（自动修复）
ruff check backend/ --fix

# 类型检查
mypy backend/core/ backend/personal/

# 运行测试
pytest backend/tests/ -v

# 确认无敏感信息
git diff --cached | grep -i "token\|password\|secret"
```

### 6.2 API 调用约定

```typescript
// frontend/personal/lib/api.ts
const API_BASE = process.env.NEXT_PUBLIC_API_URL;

export async function fetchHotItems(params: HotItemsParams) {
  const query = new URLSearchParams(params as any).toString();
  const res = await fetch(`${API_BASE}/hot-items?${query}`);
  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json() as Promise<HotItemsResponse>;
}
```

## 7. 测试策略

### 7.1 测试金字塔

```
           ┌──────────┐
           │  E2E     │  ← 少量（可选，M3 考虑）
           │  手动验收 │
          ┌┴──────────┴┐
          │  集成测试   │  ← 关键路径：采集 → DB → API
         ┌┴────────────┴┐
         │   单元测试    │  ← 大量：normalizer、validator、rank_change
         └──────────────┘
```

### 7.2 后端测试规范

```bash
# 运行全部
pytest backend/tests/ -v

# 运行单个模块
pytest backend/tests/core/test_link_validator.py -v

# 带覆盖率
pytest backend/tests/ --cov=backend/core --cov-report=term-missing
```

**关键原则**：
- 采集器测试**必须 mock HTTP**，禁止真实请求
- 每个采集器至少覆盖：正常返回、空结果、异常三种场景
- 数据库测试使用 `sqlite:///:memory:`

### 7.3 各里程碑测试重点

| 里程碑 | 必须测试 |
|--------|----------|
| M0 | GitHub 采集器 mock、normalizer 映射、URL 白名单、API 响应格式 |
| M1 | 微博/百度采集器 mock、多平台隔离（一个失败不影响另一个） |
| M2 | rank_change 计算、本周聚合逻辑、链接校验 HEAD mock |
| M3 | 搜索过滤、导出格式正确性 |

## 9. 常见问题

### Q1: 后端 import core 报 ModuleNotFoundError

设置 `PYTHONPATH` 包含 `backend/` 根目录。Docker 中已通过 Dockerfile 的 `ENV PYTHONPATH` 处理。

### Q2: GitHub API 限流 (403)

配置 `GITHUB_TOKEN` 环境变量。未认证每小时 60 次，Token 认证后 5000 次。

### Q3: 微博采集失败

预期行为。微博反爬严格，采集器会：
1. 记录错误到 `platform_health`
2. API 返回上次成功缓存
3. 前端展示「数据源暂不可用」

### Q4: 前端页面白屏

检查 `NEXT_PUBLIC_API_URL` 是否正确指向后端。开发模式下后端默认 `:8000`，前端 `:3000`。
