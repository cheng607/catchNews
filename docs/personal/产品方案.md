# CatchNews 个人版 — 产品方案

> 版本：v1.0  
> 更新日期：2026-06-14  
> 定位：个人用户的「多维度热点速览台」—— 快速、可靠、可跳转

---

## 1. 产品概述

### 1.1 一句话描述

CatchNews 个人版是一个聚合 **大众娱乐、技术、新闻** 等多赛道热点的轻量工具，帮助用户在单一界面内快速获取 **本周 / 实时** 热点，并通过 **官方来源跳转链接** 直达原文，确保信息真实可靠。

### 1.2 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 热点分散在微博、GitHub、知乎等多个平台，切换成本高 | 多源聚合，按赛道分类展示 |
| 二手转载信息失真、链接失效 | 只存官方榜单/官方页面 URL，带来源标识 |
| 不知道「现在什么在火」 | 实时榜 + 本周汇总，一屏看清 |
| 技术向和娱乐向需求不同，混在一起难找 | 赛道 Tab：娱乐 / 技术 / 综合 |

### 1.3 产品边界（个人版不做的事）

- 不提供复杂 BI 报表、自定义 SQL、多租户管理
- 不做全文爬取与内容存储（仅榜单元数据 + 跳转链接）
- 不做 AI 深度解读、舆情预警（留给产品版）
- 不做用户注册与云端同步（本地使用为主，可选导出）

## 2. 目标用户

### 2.1 用户画像

| 画像 | 描述 | 典型场景 |
|------|------|----------|
| **信息型读者** | 关注社会热点、娱乐八卦 | 午休刷微博热搜 equivalent |
| **开发者** | 关注 GitHub Trending、Star 暴涨项目 | 周末看有什么新项目值得 Star |
| **跨界用户** | 既关心娱乐也关心技术 | 一个 App/页面搞定两类需求 |

### 2.2 使用频率

- **高频**：每日 1–3 次，每次 3–10 分钟（刷实时榜）
- **中频**：每周 1 次，查看「本周热点汇总」

## 3. 数据源规划

### 3.1 赛道与平台

| 赛道 | 平台 | 采集内容 | 可靠性策略 | 优先级 |
|------|------|----------|------------|--------|
| **大众娱乐** | 微博热搜 | 排名、标题、热度值、链接 | 链接指向 `s.weibo.com` 官方热搜 | P0 |
| **大众娱乐** | 百度热搜 | 排名、标题、指数、链接 | 链接指向 `top.baidu.com` | P0 |
| **大众娱乐** | 知乎热榜 | 排名、标题、热度、链接 | 链接指向 `zhihu.com/hot` 对应条目 | P1 |
| **大众娱乐** | B站热门 | 排名、标题、播放量、链接 | 链接指向 `bilibili.com` 官方 | P2 |
| **技术向** | GitHub Trending | 仓库名、语言、Star 数、今日 Star、链接 | 链接指向 `github.com/{owner}/{repo}` | P0 |
| **技术向** | GitHub Star 增长榜 | 24h/7d Star 增量 Top N | 同上，数据来自 GitHub API 或 Trending 页 | P0 |
| **技术向** | Hacker News | 排名、标题、分数、链接 | 链接指向 `news.ycombinator.com` | P1 |
| **技术向** | Product Hunt | 当日产品、Upvote、链接 | 链接指向 `producthunt.com` | P2 |
| **综合新闻** | RSS（36氪、虎嗅等） | 标题、发布时间、链接 | 仅白名单 RSS 源 | P1 |

### 3.2 数据字段（统一最小模型）

每条热点记录包含：

```
{
  "id": "唯一 ID（平台 + 平台内 ID/hash）",
  "platform": "weibo | github | zhihu | ...",
  "track": "entertainment | tech | news",
  "title": "展示标题",
  "url": "官方跳转链接（必填，需校验可达）",
  "rank": "当前排名（nullable）",
  "heat_score": "平台原始热度值（nullable）",
  "metrics": {
    "stars": "GitHub Star 总数",
    "stars_today": "今日新增 Star",
    "stars_delta_7d": "7 日 Star 增量",
    "search_index": "搜索指数",
    "read_count": "阅读量",
    "...": "平台特有字段，原样保留"
  },
  "captured_at": "抓取时间 ISO8601",
  "first_seen_at": "首次出现在榜时间"
}
```

### 3.3 真实可靠保障机制

1. **链接白名单**：只允许跳转至各平台官方域名（见附录 A）
2. **来源标签**：每条展示「来自微博热搜」「来自 GitHub Trending」
3. **抓取时间戳**：显示「更新于 X 分钟前」，超过阈值标灰提示可能过期
4. **不二次加工标题**：展示原文标题，不做 LLM 改写（避免失真）
5. **链接存活检测**：定时 HEAD 请求，失效链接标记「链接可能已失效」并隐藏或降权
6. **GitHub 数据**：优先 GitHub REST API（Star 数为官方数据），Trending 页作补充

## 4. 功能需求

### 4.1 信息架构

```
首页
├── 赛道 Tab：全部 | 娱乐 | 技术 | 新闻
├── 时间 Tab：实时 | 今日 | 本周
└── 平台筛选（可选）：微博 / GitHub / 知乎 / ...

详情（可选，轻量）
└── 点击条目 → 新标签页打开官方链接（默认行为）
```

### 4.2 功能清单

#### F1 实时热点榜（P0）

- 按赛道展示各平台当前 Top 榜（默认 Top 20）
- 每条展示：排名、标题、核心指标（热度/Star/增量）、来源标签、更新时间
- 点击整行 → 跳转官方 URL
- 支持手动刷新 + 自动刷新（默认 15–30 分钟，可配置）

#### F2 本周热点汇总（P0）

- 聚合过去 7 天内「曾进入 Top N」的条目
- 按「出现天数」「平均排名」「最高排名」简单排序
- 标注：本周新上榜 / 持续在榜
- GitHub 专项：本周 Star 增长最多的项目 Top 10

#### F3 技术向专项卡片（P0）

- **GitHub Trending 今日**：按语言筛选（All / Python / JavaScript / Go…）
- **Star 增长榜**：24h / 7d 增量排序
- 展示：⭐ 总数、📈 今日 +N、📈 7日 +N
- 一键跳转仓库

#### F4 娱乐向专项卡片（P0）

- 微博 / 百度 热搜并排或 Tab 切换
- 展示：排名变化箭头（↑3 ↓2 NEW）
- 热度值或搜索指数（有则展示）

#### F5 搜索（P1）

- 在当前已抓取数据中按标题关键词过滤
- 不支持全网搜索（个人版范围）

#### F6 收藏 / 稍后读（P1）

- 本地收藏条目（LocalStorage / SQLite）
- 导出为 Markdown 或 CSV

#### F7 数据导出（P2）

- 导出当前视图为 CSV / Markdown
- 含：标题、链接、排名、指标、抓取时间

## 5. 量化指标（个人版展示层）

个人版 **展示** 平台原生量化数据，**不做** 复杂二次建模。

| 指标 | 适用平台 | 展示方式 |
|------|----------|----------|
| 排名 `rank` | 微博、百度、知乎、HN | #1 #2 … + 变化箭头 |
| 热度值 / 搜索指数 | 微博、百度 | 原始数值，格式化（1.2M） |
| Star 总数 | GitHub | ⭐ 12,345 |
| Star 今日增量 | GitHub | +892 today |
| Star 7日增量 | GitHub | +3,200 / 7d |
| 出现天数 | 全部（本周视图） | 「在榜 5 天」 |
| 平均排名 | 全部（本周视图） | 「均 rank 8.2」 |

## 6. 非功能需求

| 类别 | 要求 |
|------|------|
| **性能** | 首屏加载 < 2s（本地缓存命中时） |
| **可用性** | 单平台抓取失败不影响其他平台展示 |
| **更新频率** | 娱乐榜 15–30 min；GitHub 1–6 h（受 API 限制） |
| **离线** | 展示最近一次成功抓取的缓存数据 |
| **隐私** | 无账号、无上传用户行为（可选匿名统计） |
| **部署** | 支持本地 Docker 一键启动 / 桌面轻量版 |

## 7. 技术方案概要

### 7.1 架构

```
Scheduler → Collectors（各平台适配器）→ Normalizer → SQLite/PostgreSQL
                                                      ↓
                                              Static JSON Cache
                                                      ↓
                                              Web UI（Streamlit / Next.js）
```

### 7.2 采集策略

| 平台 | 方式 | 频率 |
|------|------|------|
| GitHub Trending | 爬取 trending 页 或 gh CLI / 第三方 | 6h |
| GitHub Star | REST API `/repos/{owner}/{repo}` | 随 Trending 条目 |
| 微博热搜 | 第三方 API 或 Playwright | 30min |
| 百度热搜 | HTTP JSON 接口 | 30min |
| 知乎热榜 | HTTP JSON 接口 | 30min |

### 7.3 个人版推荐技术栈

- **后端**：Python + FastAPI（或纯脚本 + SQLite）
- **调度**：APScheduler
- **前端**：Streamlit（最快）或 Next.js + Tailwind（体验更好）
- **存储**：SQLite（够用）或 PostgreSQL
- **部署**：Docker Compose 单机

## 8. 里程碑

| 阶段 | 时间 | 交付物 |
|------|------|--------|
| **M0** | 第 1 周 | GitHub Trending + Star 指标；本地 Web 列表 + 跳转 |
| **M1** | 第 2 周 | 微博 + 百度热搜；赛道 Tab；实时/本周切换 |
| **M2** | 第 3 周 | 链接校验、缓存、自动刷新、排名变化 |
| **M3** | 第 4 周 | 收藏、导出、知乎热榜；Docker 打包 |

### 8.1 M0 验收标准

- [ ] GitHub Trending 至少 2 个时间维度（daily / weekly）
- [ ] 每条显示 Star 总数 + 今日/周期增量（有数据则展示）
- [ ] 点击跳转 GitHub 官方仓库页
- [ ] 数据本地持久化，刷新页面不丢失

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 微博等平台反爬 | 娱乐榜不可用 | 备用源（百度/知乎）；降低频率；标注「数据源暂不可用」 |
| GitHub Trending 无官方 API | 解析不稳定 | 多 parser 版本；Star 数据走官方 API 保底 |
| 链接失效 | 体验差 | 定时校验 + 降权隐藏 |
| 热点标题敏感 | 合规 | 仅展示榜单元数据，不存全文；个人自用 |
