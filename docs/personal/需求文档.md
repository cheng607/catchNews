## 1. 文档说明

### 1.1 目的

将产品方案转化为可执行的开发规格，明确 **技术栈、模块划分、接口契约、数据模型、功能优先级与验收标准**，供前后端与采集模块并行开发。

### 1.2 读者

- 开发（前端 / 后端 / 采集）
- 自测与验收
- 后续产品版扩展时的共享层参考

### 1.3 术语

| 术语 | 含义 |
|------|------|
| 赛道 (track) | entertainment / tech / news |
| 平台 (platform) | weibo、github、baidu 等具体数据源 |
| 热点条目 (item) | 一条榜单元数据 + 官方跳转 URL |
| 采集器 (collector) | 对接单一平台的适配器 |
| 快照 (snapshot) | 某时刻某平台 Top N 的完整抓取结果 |

## 2. 项目目标

### 2.1 一句话

构建个人用户的「多维度热点速览台」：聚合娱乐、技术、新闻多源榜单，展示实时 / 本周热点，点击跳转官方原文，数据本地持久化，单机 Docker 可运行。

### 2.2 核心成功指标（开发需可度量）

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| 首屏加载 | < 2s（缓存命中） | Lighthouse / 手动计时 |
| 链接有效跳转率 | > 98% | 链接校验任务统计 |
| 数据源可用率 | > 95%（单源失败可降级） | 采集健康检查 |
| 找到目标热点 | < 30s | 可用性走查 |

### 2.3 明确不做（个人版边界）

- 用户注册、云端同步、多租户
- 全文爬取与正文存储
- AI 解读、舆情预警、自定义监控告警
- 复杂 BI、SQL 查询、历史趋势图（留给产品版）
- 全网搜索（仅本地已抓取数据过滤）

## 3. 技术栈选型

### 3.1 推荐方案（默认采用）

| 层级 | 技术 | 版本建议 | 选型理由 |
|------|------|----------|----------|
| 语言 | Python 3.11+ | 3.11 | 采集生态成熟，与 APScheduler 配合简单 |
| Web 框架 | FastAPI | 0.110+ | 轻量 REST API，OpenAPI 自动生成 |
| 任务调度 | APScheduler | 3.10+ | 单机 cron 足够，无需 Celery |
| 数据库 | SQLite | 3.x | 个人版数据量小，零配置 |
| ORM | SQLAlchemy 2.x | 2.0+ | 模型清晰，后续可迁 PostgreSQL |
| HTTP 客户端 | httpx | 0.27+ | 异步友好，链接 HEAD 校验 |
| HTML 解析 | selectolax / BeautifulSoup4 | — | GitHub Trending 页解析 |
| GitHub API | PyGithub 或 httpx 直调 | — | Star 数为官方数据 |
| 浏览器自动化（备用） | Playwright | — | 仅微博等 HTTP 失败时启用 |
| 前端 | **Next.js 14+ (App Router)** | 14+ | 体验优于 Streamlit |
| UI | Tailwind CSS + shadcn/ui | — | 组件复用高 |
| 本地收藏 | 浏览器 LocalStorage + 可选 SQLite 同步 | — | P1 收藏可先纯前端 |
| 容器化 | Docker + Docker Compose | — | 一键本地启动 |
| 配置 | pydantic-settings + `.env` | — | 刷新间隔、API Token 等 |

### 3.2 备选方案

| 场景 | 备选 | 说明 |
|------|------|------|
| 极速验证 M0 | Streamlit 单页 | 1–2 天可出 Demo，M1 前建议迁移 Next.js |
| 数据量增大 | PostgreSQL | 与产品版共享采集层时替换 SQLite |
| 桌面离线 | Tauri + Next.js 静态导出 | P2，非首期范围 |

### 3.3 待确认项（建议默认）

| 问题 | 建议默认 | 影响 |
|------|----------|------|
| 前端 Streamlit vs Next.js | **Next.js** | 目录结构、部署方式 |
| 移动端适配 | **M3 做响应式（P2）** | CSS 断点规划 |
| 微博数据源 | **先 HTTP/第三方 JSON，失败降级 Playwright** | 依赖与 Docker 镜像体积 |
| 部署形态 | **本地 Docker Compose 为主** | 无公网也可跑调度 |

## 5. 数据模型

### 5.1 热点条目（逻辑模型）

与产品文档一致，API 与 DB 均使用该结构：

```json
{
  "id": "github:anthropics/claude-code",
  "platform": "github",
  "track": "tech",
  "title": "anthropics/claude-code",
  "url": "https://github.com/anthropics/claude-code",
  "rank": 1,
  "heat_score": null,
  "metrics": {
    "stars": 12400,
    "stars_today": 892,
    "stars_delta_7d": 3200,
    "language": "TypeScript"
  },
  "rank_change": { "type": "up", "value": 1 },
  "captured_at": "2026-06-14T10:30:00+08:00",
  "first_seen_at": "2026-06-10T08:00:00+08:00",
  "link_status": "ok",
  "source_label": "GitHub Trending"
}
```

### 5.2 数据库表设计

#### `hot_items`（当前最新榜）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | `{platform}:{platform_item_id}` |
| platform | TEXT | weibo / github / baidu / … |
| track | TEXT | entertainment / tech / news |
| title | TEXT | 原文标题，不改写 |
| url | TEXT | 官方链接，必填 |
| rank | INTEGER NULL | 当前排名 |
| heat_score | REAL NULL | 平台原始热度 |
| metrics | JSON | 扩展指标 |
| captured_at | DATETIME | 最近抓取时间 |
| first_seen_at | DATETIME | 首次上榜 |
| link_status | TEXT | ok / stale / broken |
| is_active | BOOLEAN | 是否在当前榜展示 |

#### `hot_snapshots`（历史快照，供本周聚合与排名变化）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增 |
| platform | TEXT | |
| snapshot_at | DATETIME | 快照时间 |
| payload | JSON | Top N 完整列表 |

#### `platform_health`（采集健康）

| 字段 | 类型 | 说明 |
|------|------|------|
| platform | TEXT PK | |
| last_success_at | DATETIME | |
| last_error | TEXT NULL | |
| status | TEXT | ok / degraded / down |

### 5.3 ID 生成规则

```
id = f"{platform}:{normalized_key}"
```

| 平台 | normalized_key 示例 |
|------|---------------------|
| github | `{owner}/{repo}` 小写 |
| weibo | 热搜词 hash 或官方条目 ID |
| baidu | 条目 ID 或 title hash |

### 5.4 链接白名单（强制校验）

写入与跳转前校验 URL host 属于附录 A 域名列表；否则拒绝入库并在 API 层过滤。

```
weibo.com, s.weibo.com
top.baidu.com, www.baidu.com
zhihu.com, www.zhihu.com
github.com, www.github.com
news.ycombinator.com
producthunt.com
bilibili.com, www.bilibili.com
（RSS 源各自官方域名）
```

## 6. 采集层需求

### 6.1 采集器接口（抽象）

每个 collector 实现：

```python
class BaseCollector(ABC):
    platform: str
    track: str
    default_top_n: int = 20

    async def fetch(self) -> list[RawItem]: ...
    def normalize(self, raw: RawItem) -> HotItem: ...
```

### 6.2 平台实现规格

| 平台 | 优先级 | 采集方式 | 频率 | Top N | track |
|------|--------|----------|------|-------|-------|
| GitHub Trending | P0 | 爬 trending 页 + 解析 | 6h | 20 | tech |
| GitHub Star 增量 | P0 | REST API `/repos/{o}/{r}` | 随 Trending | — | tech |
| 微博热搜 | P0 | HTTP JSON / Playwright 备用 | 30min | 20 | entertainment |
| 百度热搜 | P0 | HTTP JSON | 30min | 20 | entertainment |
| 知乎热榜 | P1 | HTTP JSON | 30min | 20 | entertainment |
| Hacker News | P1 | 官方 API / 页面 | 1h | 20 | tech |
| RSS（36氪、虎嗅等） | P1 | feedparser | 1h | 20 | news |
| B站热门 | P2 | HTTP | 30min | 20 | entertainment |
| Product Hunt | P2 | HTTP / API | 6h | 10 | tech |

### 6.3 采集失败策略

1. 单平台异常 **不阻塞** 其他平台调度
2. 写入 `platform_health`，前端展示「数据源暂不可用」
3. API 返回该平台上一次成功快照（`cache/` JSON 兜底）
4. 连续失败 3 次标记 `down`，成功一次恢复 `ok`

### 6.4 排名变化计算

- 对比当前榜与 **上一快照** 同 platform 条目 rank
- 输出：`up` / `down` / `new` / `flat` + `value`（位次变化绝对值）
- 新条目（id 首次出现）标记 `new`

### 6.5 链接存活检测

- 定时任务：对已入库 URL 发 HEAD（超时 5s，跟随重定向）
- `broken`：标记 `link_status=broken`，列表默认隐藏，可通过筛选「含失效链接」查看
- 娱乐榜 URL：每 6h 批量校验；GitHub：每周校验

## 7. API 设计

Base URL: `http://localhost:8000/api/v1`

### 7.1 热点列表

```
GET /hot-items
```

| 参数 | 类型 | 说明 |
|------|------|------|
| track | string | all / entertainment / tech / news |
| time_range | string | realtime / today / week |
| platform | string | 可选，逗号分隔 |
| top_n | int | 默认 20，最大 50 |
| q | string | P1 标题关键词 |
| include_broken | bool | 默认 false |

**响应**

```json
{
  "items": [ /* HotItem[] */ ],
  "meta": {
    "updated_at": "2026-06-14T10:30:00+08:00",
    "platforms": {
      "github": { "status": "ok", "updated_at": "..." },
      "weibo": { "status": "degraded", "updated_at": "..." }
    }
  }
}
```

### 7.2 本周汇总

```
GET /weekly-summary
```

| 参数 | 类型 | 说明 |
|------|------|------|
| track | string | 可选 |
| top_n | int | 默认 20 |

**响应字段（每条）**

- `title`, `url`, `platform`, `source_label`
- `days_on_chart`：在榜天数
- `avg_rank`, `best_rank`
- `badge`：`new_this_week` | `persistent`
- GitHub 专项：`stars_delta_7d`

### 7.3 GitHub 专项

```
GET /github/trending?language=all|python|javascript|go
GET /github/star-growth?period=24h|7d&top_n=10
```

### 7.4 健康与刷新

```
GET /health          # 各 platform 状态
POST /refresh        # 手动触发全量或单 platform 采集
GET /refresh/status  # 可选：异步任务进度
```

### 7.5 导出（P2）

```
GET /export?format=csv|markdown&track=&time_range=
```

## 10. 非功能需求

| 类别 | 要求 | 实现要点 |
|------|------|----------|
| 性能 | 首屏 < 2s | JSON 缓存 + DB 索引 `(platform, captured_at)` |
| 可用性 | 单源失败不影响其他源 | 采集隔离 try/except |
| 更新频率 | 娱乐 15–30min；GitHub 1–6h | scheduler cron |
| 离线 | 展示最近成功缓存 | `cache/{platform}.json` |
| 隐私 | 无账号、无行为上传 | 无第三方 analytics |
| 安全 | URL 白名单、无 SSRF | link_validator |
| 日志 | 结构化 logging | platform、duration、error |
| 配置 | `.env.example` 文档化 | GITHUB_TOKEN 可选提高限额 |

## 12. 开发里程碑与任务拆解

### M0 — 第 1 周：GitHub 闭环

**目标**：技术向最小可用

| 任务 | 负责模块 | 产出 |

## 14. 风险与依赖

| 风险 | 影响 | 应对 | 负责人 |
|------|------|------|--------|
| 微博反爬 | 娱乐 P0 受阻 | 百度优先；Playwright；标注不可用 | 采集 |
| GitHub Trending 无 API | 解析脆 | 多 parser；Star 走 API | 采集 |
| GitHub API 限流 | Star 补全失败 | GITHUB_TOKEN；降频 | 后端 |
| 链接失效 | 体验差 | HEAD 校验 + 隐藏 | 后端 |
| 标题敏感内容 | 合规 | 仅榜单 metadata，个人自用 | 产品 |

**外部依赖**

- GitHub REST API（可选 Token）
- 各平台公开榜单页面或 JSON 接口（可能变更，需监控）
